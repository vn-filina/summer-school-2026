# Задача 3. Разработка функционального бэкенд-прототипа (FastAPI, SQLAlchemy, Pydantic и SQLite)

### 1. Симптом / Цель
Ранее спроектированные аналитические требования и схемы данных существовали только в виде документации. Для проверки жизнеспособности MVP гончарной мастерской «Глина» необходимо было создать полноценный рабочий прототип серверной части. Цель — совместно с ИИ-ассистентом разработать бэкенд на FastAPI, настроить реляционную базу данных SQLite через ORM SQLAlchemy, внедрить строгую валидацию данных через Pydantic и написать ядро бизнес-логики (сидирование, блокировка овербукинга, идемпотентность, правила отмены).

### 2. Требования
Разрабатываемый прототип бэкенда должен аппаратно исполнять все заложенные правила:
* **Стек технологий:** FastAPI (REST API), Python 3.10+, SQLAlchemy (ORM), Pydantic v2 (валидация), SQLite (БД).
* **Бизнес-логика бэкенда:** Сквозная автоматическая генерация расписания (демо-данные на 180 дней вперед), атомарная проверка доступных мест для исключения овербукинга (FR-004), динамический расчет стоимости (FR-006) и разделение логики отмены по правилу 3 часов (FR-007, FR-008).

### 3. Реализация и доработка
Вместе с ИИ я написала и отладила программный код серверной части, разделив его на 4 логических модуля:

1. `db.py` — конфигурация подключения к SQLite (`pottery_clay.db`) с поддержкой многопоточности (`check_same_thread=False`). Описаны декларативные модели: `UserDB`, `SlotDB` (с property-методом `base_price`), `BookingDB` и `IdempotencyKeyDB`.
2. `schemas.py` — схемы Pydantic для входящих запросов и ответов (DTO). Включает в себя регулярные выражения для очистки телефонов от мусорных символов и пробелов, предотвращая появление дубликатов.
3. `crud.py` — ядро системы. Реализована функция `seed_demo_data` (автоматически наполняет базу МК от мастеров Дмитрия, Марины, Елены и Алексея), алгоритм `create_booking` с защитой от конкурентной записи и уникальными UUID ключами идемпотентности, а также функция отмены с расчетом временной разницы `time_diff`.
4. `main.py` — точка входа. Настроены эндпоинты для симуляции СМС-авторизации (`/auth/send-code`, `/auth/verify-code`), выдача актуального расписания на 7 дней (`/slots`) и эндпоинты для управления бронированиями (`/bookings`, `/bookings/me`).

### 4. Отправленные ИИ промпты (Мои запросы к ассистенту)
> **Мой запрос 1:** «Напиши файл db.py для нашего проекта гончарной мастерской. Используй SQLAlchemy и SQLite. Создай таблицы для пользователей, слотов и бронирований в строгом соответствии с нашей data-model. Добавь вычисляемое свойство base_price для слота: если программа "Ручная лепка", то 1500, иначе 2000.»
> 
> **Мой запрос 2:** «Теперь давай создадим schemas.py на Pydantic. Напиши схемы для запросов авторизации, создания брони и ответов сервера. В схеме SendCodeRequest напиши жесткий валидатор, который очистит строку от скобок, дефисов и пробелов через регулярки, проверит длину и вернет чистый номер.»
> 
> **Мой запрос 3:** «Самая важная часть — crud.py. Напиши функцию seed_demo_data, которая при старте проверяет базу и, если она пустая, генерирует расписание на полгода (180 дней) вперед с чередованием мастеров (Дмитрий, Марина, Елена, Алексей) и программ. Также напиши create_booking: сервер должен проверять available_places, уменьшать их на 1 при записи, рассчитывать final_price и сохранять Идемпотентный ключ, чтобы два одинаковых запроса не создали две брони.»

> **Мой запрос 4:** «Собери все это в файле main.py с использованием FastAPI. Настрой событие для автоматического создания таблиц и вызова сидирования данных. Напиши эндпоинты GET /slots, POST /bookings и POST /bookings/{id}/cancel.»

### 5. Ручная проверка (Тестирование)
Я проверила работоспособность бэкенда с помощью встроенного интерактивного окружения:
1. Запустила локальный сервер командой `uvicorn main:app --reload`. Логи подтвердили успешный запуск, создание файла `pottery_clay.db` и автоматическое выполнение сидирования (наполнение базы слотами).
2. Открыла в браузере интерактивную документацию Swagger UI по адресу `http://127.0.0.1:8000/docs`.
3. Выполнила тестовый запрос к `GET /slots`. Бэкенд успешно вернул JSON-массив сгенерированных занятий, где для ручной лепки лимит мест равен 6, а для гончарного круга — 10. Все поля (`base_price`, `master_name`, `start_time`) отдаются корректно. Слой работы с данными функционирует без ошибок.
