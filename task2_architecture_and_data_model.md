# Задача 2. Проектирование архитектуры системы и модели данных (API, Sequence-диаграммы и ER-структура)

### 1. Симптом / Цель
Отсутствие строгого технического описания взаимодействия между клиентской частью (Streamlit) и серверной частью (FastAPI), а также отсутствие нормализованной схемы базы данных. Без этого этапа проектирования была высока вероятность рассинхронизации разработки фронтенда и бэкенда, возникновения багов с типами данных и овербукингом. Цель — совместно с ИИ спроектировать надежную реляционную модель данных (ER) и зафиксировать протокол сетевого взаимодействия (API Contract) с защитой от сбоев связи (идемпотентность).

### 2. Требования
Проектирование должно обеспечивать аппаратную реализацию ранее собранных бизнес-правил:
* **Трассировка требований:** Сущности БД должны поддерживать лимиты мест (FR-004), хранение канонических телефонов (FR-003) и расчет цен с прокатом (FR-006).
* **Надежность связи:** Проектирование механизма `Idempotency-Key` для безопасного повтора POST-запросов на бронирование при нестабильном мобильном интернете у клиента.

### 3. Реализация и доработка
В ходе совместных сессий проектирования с ИИ-ассистентом я разработала техническую архитектуру проекта и зафиксировала её в каталоге `02-architecture-design/4-design/`:

1. `data-model.md` — детальное маркдаун-описание логической структуры базы данных. Спроектированы 3 ключевые таблицы для SQLite (`slots`, `bookings`, `users`) и вспомогательная таблица `idempotency_keys`. Описаны все типы полей (UUID, DateTime, Boolean, Enum), связи типа «Один-ко-многим» (1 Slot : N Bookings) и триггерные ограничения (например, валидация остатка мест `available_places`).
2. `api-sequence.md` — пошаговая Sequence-спецификация API-взаимодействия для основного сценария использования (UC-1 «Запись на МК»). Документ описывает точную последовательность вызовов: `GET /slots` (получение сетки расписания) $\rightarrow$ `POST /bookings` (отправка JSON-тела брони вместе с UUID-ключом в заголовках) $\rightarrow$ Ответ сервера `201 Created`.

### 4. Отправленные ИИ промпты (Мои запросы к ассистенту)
> **Мой запрос 1:** «Теперь давай спроектируем реляционную модель данных для SQLite. Напиши структуру файла data-model.md. Мне нужны таблицы для слотов расписания, пользователей и самих бронирований. Пропиши типы полей (UUID, String, DateTime, Boolean, Enum), их назначение и бизнес-ограничения. И не забудь добавить сущность для хранения ключей идемпотентности.»
> 
> **Мой запрос 2:** «Схема данных отличная. Теперь помоги расписать контракт сетевого взаимодействия (спецификацию API) в файле api-sequence.md. Опиши пошагово в виде текстовой sequence-диаграммы сценарий успешной записи клиента (UC-1). Какие эндпоинты вызываются (GET, POST), какие структуры JSON летят в запросе и ответе, и как именно в заголовках передается Idempotency-Key для защиты от дубликатов.»

### 5. Ручная проверка (Тестирование)
Я проверила разработанные архитектурные чертежи на предмет соответствия реальному коду приложения:
* **Валидация ER-модели:** Проверила файлы `db.py` и `schemas.py` в PyCharm. Убедилась, что декларативные модели SQLAlchemy (`SlotDB`, `BookingDB`, `UserDB`, `IdempotencyKeyDB`) полностью повторяют поля, типы данных и внешние ключи (Foreign Keys), заложенные в `data-model.md`.
* **Валидация API-контракта:** Открыла интерактивную документацию бэкенда Swagger UI (`http://127.0.0.1:8000/docs`) при запущенном сервере и сверила структуру параметров `POST /bookings` и `GET /slots`. Контракт бэкенда полностью совпадает со спроектированной sequence-схемой.
